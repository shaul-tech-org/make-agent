# 라우팅 결정 상세 기준

## 2단계 라우팅 알고리즘

### 1단계: 질문 여부 판별

"설명", "뭐야", "알려줘" 등 단순 질문 패턴 → 직접 응답 (에이전트 호출 생략)

### 2단계: 복잡도 판단 + 에이전트 선택

**단순 (에이전트 1개):**
- 단일 기술 영역만 필요
- 작업 분해 불필요
- 예: "로그인 API 만들어줘", "버튼 색상 바꿔줘"

**복합 (에이전트 2+):**
- 여러 기술 영역에 걸침
- 작업 분해 + 의존성 관리 필요
- 예: "사용자 관리 기능", "게시판 만들어줘"

## 키워드 가중치 라우팅

| 에이전트 | 키워드 (가중치) |
|---------|---------------|
| be-developer | API(2), DB(3), 서버(2), 백엔드(3), 마이그레이션(3), FastAPI(3) |
| fe-developer | UI(2), 프론트(3), 디자인(2), 컴포넌트(3), CSS(2), React(3) |
| cto | 아키텍처(3), 기술 선택(3), 리뷰(2), 스택(2) |
| researcher | 조사(3), 분석(2), 비교(3), 리서치(3), 트렌드(2) |
| qa-engineer | 테스트(3), QA(3), 검증(3), 품질(2) |
| infra-engineer | Docker(3), CI(2), 배포(3), 인프라(3) |
| ceo | 계획(2), 우선순위(2), 로드맵(3), 전략(3) |

## 액션 오버라이드 (동사 우선)

특정 동사가 포함되면 키워드 가중치보다 우선한다:

| 동사 | 에이전트 | 예시 |
|------|---------|------|
| 배포 | infra-engineer | "API 서버 배포해줘" → infra (API 키워드 무시) |
| 테스트 | qa-engineer | "로그인 기능 테스트해줘" → qa |
| 검증 | qa-engineer | "API 검증해줘" → qa |
| 조사/리서치 | researcher | "Redis 조사해줘" → researcher |
| 비교 | researcher | "DB 비교해줘" → researcher |

## 동점 처리

키워드 점수가 동일할 경우:
- 키워드 수가 적은 쪽 = 더 전문적 = 승리
- 그래도 동점이면 be-developer 우선 (가장 일반적인 요청)
